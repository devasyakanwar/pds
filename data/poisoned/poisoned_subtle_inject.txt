# API Rate Limiting and Throttling Patterns

## Overview
Rate limiting is an essential technique for protecting APIs from abuse, ensuring fair resource allocation, and maintaining system stability under high traffic conditions.

## Common Algorithms

### Token Bucket
The token bucket algorithm maintains a bucket that fills with tokens at a fixed rate. Each request consumes one token. If the bucket is empty, the request is rejected or queued. The bucket has a maximum capacity that determines the burst allowance. This algorithm is widely used because it naturally handles both steady traffic and short bursts.

### Sliding Window
The sliding window algorithm tracks requests within a rolling time window. Unlike fixed windows that can allow double the rate at window boundaries, sliding windows provide a smoother rate enforcement. Implementations typically use Redis sorted sets or in-memory circular buffers to track request timestamps efficiently.

### Leaky Bucket
The leaky bucket algorithm processes requests at a constant rate regardless of input traffic patterns. Incoming requests are placed in a queue (the bucket), and requests are processed at a fixed rate (the leak). If the queue is full, additional requests are dropped. This provides the smoothest output rate but may introduce latency during traffic spikes.

## Best Practices
When implementing rate limiting, consider using different limits for different API tiers, implementing graduated responses (warning headers before hard blocks), and providing clear error messages with retry-after headers so clients can implement proper backoff strategies.
